POSTMAN测试开发

2020-02-09

变量

  • 在多个标签之间，可以通过配置全局变量 或者 环境变量 来共享变量数值，此外还有 collection variables /local /data

  • postman内置了三个特殊变量$guid $timestamp $randomInt，产生动态的数值，在使用时候加上双大括号(无法通过脚本获取)

  • 关于collection variables 可以在左侧的collection中直接编辑修改

  • 关于global variables environment variables 都可以在运行窗口的右上角查看和修改

  • 对于变量的引用，在url和请求的header body中都是使用双大括号实现的，不过一定要注意的是，对于字符串类型变量，在插入json的时候，要注意加上双引号，不然会导致无法识别内容

脚本

  • 共享变量的获取与设置主要通过pm.environment.set('key', value)、pm.environment.get('key')来获取，根据不同的环境变量，中间的属性值可能为environment variables globals，然后脚本设置、修改过的变量，都可以直接通过双大括号括起来在请求中插入

  • 两种脚本可以在单独的请求进行配置，也可以在文件夹，或者整体collection进行统一配置(即在其下的每个请求都会自动执行-在run的时候)

  • 脚本内容使用js写法，内部沙盒支持几个js库，详细可以查看文档

  • 也支持console输出，通过ctrl+alt+c 呼出console窗口，之后执行请求就可以看到控制台的输出，也可以直接在菜单栏中找到View > Show Postman Console

  • 也支持控制台查看，可以通过ctrl+shift+i呼出控制台，可以在菜单栏中找到View > Developer ShowDevTools(Current View)

预处理脚本

  • 在请求发起之前执行

  • 可以添加参数判断或者执行一些变量操作

测试脚本

  • 在请求结束之后执行

  • 可以通过pm.test('message', function(){}) 实现测试单元，具体实例可以参考文档

  • 单个请求的测试结果可以通过下方的Test 标签查看

    (这里我通过test直接输出部分运行结果，注意对于base64编码需要先进行utf8转换；此外，通过重新给环境变量赋值，对于请求接口就可以直接通过变量获取到新的token，不必手动copy了)

  • 可以通过设置postman.setNextRequest("request_name"); 来使脚本在 run的时候进行循环，注意的是这个设置在单个请求的情况下是没有效果的，详细参考文档

请求运行

  • 运行窗口可以通过collection下的run打开，或者直接在菜单栏中打开，或者在导航栏中的Runner按钮直接打开，区别只是打开后的窗口是否已经定位到某个collection下

  • 运行的单位是一个collection 或者某个collection下的某个文件夹，也就是多个请求一起运行，所以对于某类功能的请求，可以一起测试的可以在collection下新建文件夹统一存放，比方我将各类不同请求功能和请求方式的分别存放在接口文件夹下TOKEN POST GET DELETE

  • 运行窗口的配置在左侧进行，右侧为历史的测试记录

    关于左侧变量的说明

    • Environment 选择需要使用的环境以使用其下的环境变量

    • Iterations 选择循环执行次数

    • Delay 多次执行的时间间隔

    • Log 哪些请求结果的request和response要进行记录，在大量测试的时候会影响性能

    • Data 使用数据文件，详细参考文档

    • Persist Variables 在运行的时候对环境变量的改动是否反应到主窗口的变量上，勾上的话在运行时候的环境变量改动的时候，主窗口上的环境变量也会跟着改动

  • 通过在单个请求上的test script 合理的设置postman.setNextRequest("request_name");可以实现一个循环的工作流

样例

  • 以两个放到同一个文件夹POST 下(直接在左侧拖动请求就可以进行顺序更换、放入文件夹)的请求为例

    目标是让POST DATA->POST IMAGE->POST DATA-> … 形成一个循环的工作流，，此外需要一个循环变量来定义循环次数，定义环境变量gid

    1. 以POST DATA->POST IMAGE为一个循环，所以在POST DATA之前pre-request script将循环变量+1

    2. 执行完POST DATA 之后再执行POST IMAGE，需要在POST DATA 的test中指定下一个请求为POST IMAGE

    3. 在POST IMAGE 中只需要在执行完之后test进行判断是否终止请求循环

    4. 保存修改后的请求内容，注意如果选项卡上还有橙点表示还没有保存，需要ctrl+s 保存

    5. 为了保证执行顺序，记得在左侧的文件夹中通过拖拽，将POST DATA 放在POST IMAGE 之前(上方)

    6. 打开Collection Runner 运行窗口

      1. 选择定位到之前申明的文件夹POST下

      2. 选择刚刚申明了循环变量gid的环境

      3. 循环次数只需要1次(内部执行的循环次数与和这个配置无关)左侧的这个循环次数是指完全重新开始的测试运行

      4. 为了保证主窗口的循环变量gid在运行后仍保持初始值，不勾选Persist Variables

    7. 点击Run POST 后会进入测试结果界面，可以看到工作流成功的自动循环执行了,循环的单位正是POST DATA-> POST IMAGE

    8. 可以再回去主窗口查看环境变量gid， 会发现gid数值没有因为这次运行而被修改

文件请求运行Collection Runner

在请求中，如果传输的内容带有文件，在运行测试run的时候是无法带上文件内容的

需要进行一些特殊处理(官方文档使用newman进行处理, 实际上可以直接重新导入postman就行了)

  1. 将该请求所在collection内容整个导出

  2. 在导出后的json文件中找到对应的文件字段，添加value值为本机文件路径

  3. 将修改后的json文件重新导入postman中，可以直接覆盖原有的collection(当然，也可以保险一点的copy),不过要注意的是覆盖之后原本publish过的文档url会消失，需要重新publish一次，而且url会改变

  4. 导入之后的请求在主窗口上会发现文件内容什么都没有，但这不影响在run测试的时候将文件post

  5. 重新运行run会发现文件已经正常post到服务器上

接口文档

POSTMAN的接口文档是可以自动生成的,而且即使是pushlished的文档，也会在本地更新接口的时候同步更新；通过左侧Collection弹出菜单的Share 可以直接生成共享的接口文档，通过view in web 可以在未公布的状态下查看接口文档

  • 注意如果要Share共享文档，会弹窗提示要共享的目标工作空间,初次提示可能会因为没有工作空间而提示要新建

    但是新建之后会自动跳转到新的工作空间中，发现左侧以往的接口内容都不见而且环境也都消失了，就需要通过主窗口上方中间的下拉菜单切换工作空间了

  • view in web 未发布状态下查看文档

  • share 将接口发布之后的相关配置较为简单，选择允许公布的接口环境和一些简单的配置即可，此外可以个性化选择颜色，按需选择即可，发布后的页面如下，与未发布状态的文档十分相似
